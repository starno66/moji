# 墨迹 (Moji) 用户手册

## 第一章 产品介绍

### 1.1 概述

墨迹是一款面向高校师生群体的**文档版本管理与写作协作工具**。软件以 Git 版本控制系统为核心引擎，将专业开发者使用的版本管理能力封装为直观的图形操作界面，使不具备编程背景的师生也能轻松管理文档的每一次修改记录。

在学术写作场景中，文档的反复修改是常态。一篇毕业论文从开题到定稿，往往经历数十次甚至上百次修订。传统的写作工具仅保留文档的当前状态，过往版本的找回依赖手动备份（如"论文初稿.docx""论文修改版2.docx""论文最终版.docx"），这种方式不仅效率低下，且极易造成版本混乱与内容丢失。

墨迹通过自动化的版本追踪机制，完整记录每一次文档变更的时间、作者与内容。用户可随时回溯至任意历史版本，比较不同版本之间的差异，或将特定章节恢复至指定版本。此外，墨迹内置的分支管理功能支持教师对学生文档进行独立批注与修改，经学生确认后再将修改合并至正文，实现了规范、可追溯的写作反馈流程。

### 1.2 设计理念

墨迹的设计遵循以下三项核心原则：

**以文件夹为写作单元**。软件不限制用户使用的文档格式（支持 Word、LaTeX、Markdown、纯文本等）。用户将一篇文档的各章节组织为工作区下的子文件夹，每个文件夹视为一个独立的写作单元。墨迹在此基础上提供版本追踪、历史回退与协作管理功能，而不介入具体的文档编辑过程——用户仍可使用自己熟悉的编辑器（如 Microsoft Word、TeXstudio、VS Code 等）进行写作。

**以 Git 为底层引擎**。Git 是目前全球最广泛使用的分布式版本控制系统，由 Linux 操作系统创始人 Linus Torvalds 于 2005 年创建。Git 的核心优势在于：每一次提交（Commit）都生成唯一的数字指纹，确保历史记录不可篡改；分支（Branch）的创建与合并操作极为轻量，支持多人并行工作而不产生冲突。墨迹将 Git 的命令行操作全部转化为图形界面中的按钮与菜单，用户无需学习任何 Git 命令即可享受完整的版本管理能力。

**以教学协作为目标场景**。墨迹的分支机制专为师生写作反馈场景设计：学生在主分支上撰写原文，教师从主分支创建独立的反馈分支进行批注与修改，双方的工作互不干扰。修改完成后，学生可在两个分支之间自由切换以对比查看差异，确认无误后将反馈分支合并至主分支。整个过程清晰、可追溯，避免了传统模式下"邮件发送文档 → 批注 → 发回 → 手动整合"的繁琐流程。

### 1.3 适用场景

**毕业论文与课程报告写作**。学生在撰写长篇学术文稿时，墨迹自动记录每一次修改，形成完整的写作历程日志。如遇到思路调整或内容删改，可随时回退至之前的版本，无需担心误操作导致内容永久丢失。

**教师批注与反馈**。教师为每位学生的文档创建独立反馈分支，在分支上直接进行修改与批注。学生可在原文与教师修改版之间切换查看，逐一确认修改内容后合并。整个过程无需网络连接，所有数据存储于本地。

**多人协作与合著**。小组成员各自在不同分支上撰写不同章节，定期合并至主分支。分支间的修改隔离机制保证了协作过程中不会相互覆盖对方的工作成果。

**写作进度追溯**。通过提交历史列表，学生可清晰回顾整篇文档的写作历程——每一阶段完成了哪些章节、修改集中在哪些部分、投入了多少次修订。教师也可据此评估学生的写作过程与投入程度。

### 1.4 核心特性

| 特性 | 说明 |
|:---|:---|
| 版本历史记录 | 自动追踪每次文档修改，生成带有时间戳与作者信息的提交记录 |
| 章节级回退 | 可将特定章节恢复至任意历史版本，不影响其他章节 |
| 分支管理 | 支持创建、切换、合并分支，实现师生独立协作 |
| 远程同步 | 支持推送至 GitHub 等远程仓库，实现多设备同步与云端备份 |
| AI 写作助手 | 集成大语言模型，可分析提交历史、提供写作建议与结构优化意见 |
| 格式无关 | 不限制文档格式，用户可使用任意编辑器 |
| 离线优先 | 所有版本数据存储于本地，无需网络即可完成全部版本管理操作 |
| 环境自检 | 首次启动自动检测 Git 安装状态，提供一键安装引导 |

### 1.5 技术背景：为什么选择 Git

Git 诞生于 Linux 内核开发社区。2005 年，Linux 创始人 Linus Torvalds 因原有版本管理工具的许可证变更，决定自行开发一套新的版本控制系统。他在设计之初确立了四个目标：速度足够快以处理海量提交、支持完全分布式的工作模式、确保每次提交的数据完整性、以及提供轻量级的分支操作。Git 仅用十天即完成首个可用版本，此后迅速成为全球最流行的版本控制系统，被 Google、Microsoft、Facebook 等企业广泛采用。

Git 之所以适用于写作场景，在于其核心机制与文档管理的需求高度匹配。每一次提交（Commit）实际上是对当前全部文件状态的一次完整快照，而非仅记录变化的部分，这意味着任意历史版本都可以被完整复原。分支（Branch）机制则允许在同一目录下维护多条独立的修改线，各分支之间的修改互不影响，仅在用户主动合并（Merge）时才会将差异整合。

墨迹的价值在于充当了用户与 Git 之间的翻译层。用户通过界面按钮表达意图（如"提交变更""回退至该版本""合并分支"），墨迹将其翻译为对应的 Git 命令并执行。用户所见的是文件夹、历史列表和按钮，而底层运行的是完整的 Git 版本管理系统。

| 项目 | 要求 |
